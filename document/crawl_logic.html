NetSpider 爬取逻辑详解

本文档详细解释 NetSpider 系统如何从公网爬取企业数据的实现逻辑，涵盖从用户触发搜索到数据持久化的完整链路。

二、逐层详解

第一层：BaseSpider — 抽象契约

文件：app/spiders/base.py

定义了所有爬虫必须实现的两个抽象方法：

方法输入输出用途
search()keyword: str, filters: dictlist[CompanySearchResult]根据关键词搜索，返回浅层公司列表（名称、地址、行业等）
get_detail()company: CompanySearchResultCompanyDetailResult根据搜索结果中的公司对象，抓取完整工商详情

设计意义：这是插件化的核心。新增一个真实数据源时，只需创建一个类继承 BaseSpider，实现这两个方法，然后在注册表中登记即可。路由层和调度层完全不用修改。

第二层：CrawlerManager — 调度与合并

文件：app/spiders/manager.py

search_companies() 的核心流程：

1
从 config.py 读取已启用的数据源列表（如 ["qichacha_mock"]）
2
通过 get_spider(source) 为每个数据源实例化对应的 Spider 对象
3
用 asyncio.gather() 并发请求所有数据源，而不是串行等待。假设配置了 3 个数据源，总耗时 ≈ 最慢那个源的时间，而非三者之和
4
将各来源的结果按 credit_code（统一社会信用代码）去重合并：同一公司从多个源都搜到时，只保留第一条
5
异常处理：某个数据源抛异常时，仅记录错误日志返回空列表，不影响其他源的查询结果 —— 这就是 PRD 中的降级策略

get_company_detail() 的流程更简单：根据公司记录中的 source 字段找到对应 Spider，调用其 get_detail() 方法即可。

第三层：具体 Spider 实现

文件：app/spiders/mock_spider.py

MVP 阶段使用静态数据模拟真实的网络爬取，但流程与真实爬虫完全一致：

搜索阶段（search）

1
模拟网络延迟 0.5~1.5s（asyncio.sleep(random.uniform(...))）
2
遍历 8 条预设公司数据，将关键词转为小写后，逐一匹配公司的 name、industry、address 字段
3
若有精确匹配，返回匹配子集；若无匹配，返回全部数据（模拟模糊搜索兜底行为）
4
每条结果封装为 CompanySearchResult，包含：名称、信用码、地址、行业、来源标识、来源URL

详情阶段（get_detail）

1
模拟网络延迟 0.8~2.0s
2
根据 credit_code 或公司名称在预设数据中查找
3
填充完整工商字段（法人、注册资本、成立日期、经营范围、联系方式等）返回 CompanyDetailResult
替换为真实爬虫时，这两个方法的内部实现会变成：构造目标网站的 URL（带搜索参数）→ 用 httpx 或 Playwright 发起请求 → BeautifulSoup / CSS选择器解析 HTML → 提取结构化字段填充到 Result 对象。接口签名和调用方式完全不变。

第四层：API 路由 — 串接持久化

文件：app/routers/companies.py

搜索接口 POST /api/companies/search

1
接收并校验 Pydantic SearchRequest（keyword + filters）
2
调用 search_companies() 得到 Spider 返回的去重后结果
3
写入 SearchRecord 搜索记录表（记录搜索关键词、筛选条件、结果数量、时间）
4
对每条结果：先查 DB 是否已存在相同 credit_code + source 的记录（避免重复插入），不存在则新建 Company ORM 对象
5
flush() 获取自增 ID，构建 CompanyBrief 返回前端展示
6
记录 CrawlLog：任务类型=search、状态=success/failed、耗时、错误信息

详情接口 GET /api/companies/{id}/detail

1
从 DB 查询 Company 记录（根据 URL 路径中的 id）
2
检查缓存：如果 legal_representative、registered_capital、business_scope 等关键字段已有值，直接返回 —— 不再发起网络请求
3
若无缓存数据，构造 CompanySearchResult 并调用 get_company_detail() 触发真实爬取
4
将爬取到的详情字段更新到 Company 记录中，下次访问直接走缓存
5
记录 CrawlLog（任务类型=detail）

三、关键设计决策总结

关注点实现方式所在层级
多源并发asyncio.gather() 并行查询，总耗时等于最慢数据源调度器层
结果去重用 credit_code 作为唯一键，同名同码只保留第一条调度器层
降级容错单个源 try/catch 返回空列表，不影响其他源继续执行调度器层
详情缓存首次爬取后落库，后续直接从 SQLite 读取，避免重复请求API 路由层
限频策略Mock 阶段模拟随机延迟；生产环境可通过 config 加全局 rate limiterSpider 层 / 配置
可扩展性新数据源 = 一个 Spider 类 + 注册表一行代码 + config 一行配置全局架构
审计追踪每次搜索和详情请求都写 CrawlLog，记录状态、耗时、错误API 路由层
搜索历史每次搜索写入 SearchRecord，支持后续分析热门搜索词API 路由层

五、从 Mock 到真实爬虫的迁移路径

当前 mock_spider.py 使用静态数据模拟网络请求。要对接真实数据源，仅需修改 Spider 层：

  1. 新建 Spider 文件：在 app/spiders/ 下创建如 qcc_spider.py
  2. 继承 BaseSpider：实现 search() 和 get_detail()
  3. HTTP 请求：在 search() 中用 httpx 发送带搜索参数的请求，在 get_detail() 中请求公司详情页 URL
  4. HTML 解析：用 BeautifulSoup 从响应 HTML 中提取结构化字段
  5. 注册：在 manager.py 的 SPIDER_REGISTRY 中添加一行 "qcc": QccSpider
  6. 启用：在 config.py 的 DATA_SOURCES 中添加 "qcc"

路由层、数据库层、前端界面完全不需要修改。
